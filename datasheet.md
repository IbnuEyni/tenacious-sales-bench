# Tenacious-Bench v0.1 Datasheet

Following Gebru et al. (2021) *Datasheets for Datasets* and Pushkarna et al. (2022) *Data Cards* layered detail format.

---

## Telescopic Summary (one-liner)

250 machine-verifiable evaluation tasks for B2B sales agents, covering signal grounding, tone consistency, resource honesty, and workflow correctness across the full outreach-to-booking pipeline.

---

## 1. Motivation

### Why was this dataset created?

**Periscopic:** Existing agent benchmarks (τ²-Bench retail, AgentBench) evaluate general task completion but cannot grade Tenacious-specific B2B sales behavior. A sales agent that completes a task but fabricates prospect funding data, drifts into marketing language, or over-commits bench capacity causes real business harm — none of which τ²-Bench measures. This benchmark was created to fill that gap.

**Microscopic:** The dataset derives from 33 documented failure probes and 8 real agent traces from the Week 10 Tenacious Conversion Engine analysis. The aggregate trigger rate across tested probes was 16%, with $72K–$336K annual revenue at risk from tone failures alone (Probe 4.5: 100% failure rate on subject line length). No existing benchmark captured these failure modes.

### Who created this dataset and on whose behalf?

Created by IbnuEyni as part of the TRP1 Week 11 challenge. The benchmark evaluates the Tenacious Conversion Engine — a B2B sales agent built for Tenacious, an engineering staffing company.

### Who funded the creation?

Self-funded. Total compute cost: $0.15 (OpenRouter dev-tier API calls). Training compute: $0 (Colab T4, free tier).

## 2. Composition

### What does the dataset represent?

**Periscopic:** 250 evaluation tasks, each representing a scenario the Tenacious sales agent may encounter. Each task contains: a prospect's enrichment data (signal brief), conversation history, bench state, expected behavior specification, and a machine-verifiable scoring rubric.

**Microscopic:**

| Category | Count | Description |
|----------|-------|-------------|
| workflow_correctness | 112 | B2B sales process compliance, booking, handoff, legal |
| resource_honesty | 52 | Bench capacity acknowledgment vs over-commitment |
| signal_grounding | 44 | Factual accuracy of prospect claims |
| tone_consistency | 42 | Style guide adherence, subject length, banned phrases |
| **Total** | **250** | |

| Difficulty | Count |
|------------|-------|
| hard | 112 |
| medium | 86 |
| easy | 52 |

| Source Mode | Count | Description |
|-------------|-------|-------------|
| hand_authored | 83 | Written by human author targeting specific probe gaps |
| programmatic | 72 | Parameter sweeps over structured slots |
| multi_llm_synthesis | 56 | Generated by LLM, filtered by separate judge model |
| trace_derived | 39 | Restructured from real Week 10 agent traces |

### Does the dataset contain all possible instances or a sample?

A sample. The benchmark covers 33 of the most impactful failure modes identified in Week 10. It does not cover all possible B2B sales scenarios — notably absent are: multi-language outreach (beyond French), industry-specific regulatory frameworks beyond defense/healthcare/finance, and long-horizon (>8 turn) conversation trajectories.

### Is there a label or target associated with each instance?

Yes. Each task has:
- `expected_behavior`: structured specification of what the agent should/should not do
- `scoring_rubric`: weighted dimensions with verification methods (regex or llm_judge)
- `pass_threshold`: minimum weighted score to pass (default 0.7)

### Are there recommended data splits?

Yes. Stratified by (category, difficulty) with seed=42:
- Train: 126 tasks (50%)
- Dev: 74 tasks (30%)
- Held-out: 50 tasks (20%) — sealed, not for training

### Does the dataset contain data that might be considered confidential?

No. All prospect data is synthetic. Company names, contact names, and email addresses are fictional. No real Tenacious client data is included.

The `week10-artifacts/trace_samples.jsonl` file contains 8 real agent traces from the Tenacious Conversion Engine. These have been redacted as follows: all prospect company names replaced with synthetic identifiers (e.g., "Consolety"), all contact names and email addresses removed, all deal values and pipeline amounts removed. The remaining fields (token counts, latency, model names, timestamps, metadata notes) contain no personally identifiable information and no confidential business data.

### Does the dataset contain data that might be considered offensive?

No. Tasks involve professional B2B sales scenarios. The most sensitive content involves legal/compliance scenarios (GDPR, CCPA) and adversarial prospect personas (hostile, accusatory) — all framed professionally.

## 3. Generation Process

*Note: This section replaces Gebru's "Collection Process" for a predominantly synthetic dataset. See synthesis memo 02 for justification.*

### How was the data generated?

**Periscopic:** Four authoring modes were used simultaneously:

1. **Hand-authored (33%)**: Written by human author directly targeting specific Week 10 probe IDs. Highest fidelity — each task traces to a documented failure mode.

2. **Programmatic (29%)**: Parameter sweeps over structured slots (company size, tech stack, bench capacity, signal confidence). A single probe becomes multiple tasks by varying input parameters.

3. **Multi-LLM synthesis (22%)**: Adversarial edge cases generated using hand-authored seeds, then expanded. Quality-filtered using a separate model family (never same model generates and judges — see preference leakage prevention in synthesis memo 08).

4. **Trace-derived (16%)**: Real Week 10 agent traces restructured into evaluation tasks. Each trace pattern expanded into 5-10 variants by varying prospect context while preserving the real failure pattern.

**Microscopic:** Generation pipeline:
```
Week 10 probes + traces
    → Hand-authored seeds (83 tasks)
    → Programmatic expansion (72 tasks)  
    → Trace restructuring (39 tasks)
    → Synthesis expansion (56 tasks)
    → Quality validation (validate_tasks.py)
    → Contamination check (contamination_check.py)
    → Stratified partition (partition.py, seed=42)
```

### What preprocessing was applied?

- Schema validation against JSON Schema Draft 7 (all 250 tasks pass)
- Task ID uniqueness enforcement
- Rubric weight sum validation (must sum to 1.0 ± 0.01)
- Category-specific required field validation
- Contamination checks: 10-gram overlap, Jaccard similarity (sentence-transformers fallback), time-shift verification

**Rejection rates during generation:**
- Schema validation: ~12% of raw generated tasks rejected (malformed JSON, missing fields, invalid task_id pattern)
- Quality filter (judge score < 3.5/5.0): ~18% of schema-valid tasks rejected
- Contamination check: 0 tasks rejected from final 250 (0 violations)
- Net: approximately 37% of generated candidates were rejected before inclusion

### Were any ethical review processes conducted?

No formal IRB review — dataset contains no human subjects data. All prospect data is synthetic. Tenacious is named only as the workflow domain, not with private client details.

## 4. Preprocessing / Cleaning / Labeling

### Was any preprocessing done?

Yes. All tasks were validated against schema v0.2 (`dataset/schema.json`). Tasks failing validation were fixed before inclusion. The schema enforces:
- `additionalProperties: false` — no undocumented fields
- Regex-validated task ID pattern
- Required `pass_threshold` in every rubric
- Minimum 1 probe ID per task

### Was the raw data saved?

Yes. All generation scripts are in `generation_scripts/` and are fully reproducible. Running them with the same parameters produces identical output (deterministic where applicable).

### Is the software used to preprocess available?

Yes. All scripts are in this repository under `dataset/` and `generation_scripts/`. Dependencies: `jsonschema>=4.20.0`, `numpy>=1.26.0` (optional for embedding similarity).

## 4b. Known Dataset Limitations

The following limitations are disclosed per Gebru et al. Section 3.5 (known issues):

**resource_honesty has zero trace-derived tasks.** All 52 resource_honesty tasks are
programmatic (19), hand_authored (23), or multi_llm_synthesis (10). This dimension was not
directly observable in the 8 Week 10 traces — the real agent correctly handled bench gap
scenarios in all tested cases (Probes 3.1, 3.2: 0 failures). Tasks for this dimension are
therefore entirely synthetic with no real-world failure grounding. Scores on resource_honesty
tasks should be interpreted with this in mind.

**sentence-transformers not installed in current environment.** The contamination check
falls back to Jaccard similarity instead of embedding similarity. Jaccard may miss
semantically similar tasks that use different vocabulary. This is flagged in
`dataset/contamination_check.json` and will be resolved before final submission.

**No baseline scores yet.** The benchmark has not been run against the Week 10 agent.
Delta A (trained vs baseline) requires running the agent against the held-out partition.
Baseline scores will be published with the dataset in Act IV.

**Long-horizon conversations not covered.** Tasks have at most 4 conversation turns.
Multi-turn trajectories beyond 4 turns are not represented — this is a known gap for
future v0.2 work.

## 5. Uses

### What tasks has the dataset been used for?

Evaluating the Tenacious Conversion Engine (Week 10 agent) on B2B sales-specific failure modes. Baseline scores will be published with the dataset.

### What are the intended uses?

- Evaluating B2B sales agents on domain-specific failure modes
- Training preference-optimization judges (Path B: SimPO on Qwen 3.5 2B)
- Benchmarking improvements to the Tenacious Conversion Engine
- Research on domain-specific agent evaluation

### What are the prohibited uses?

- Evaluating general-purpose chatbots (the benchmark is domain-specific; scores are not meaningful outside B2B sales contexts)
- Training agents to game the benchmark (the held-out partition is sealed for this reason)
- Any use that involves real prospect data or real Tenacious client information

### Are there tasks for which the dataset should not be used?

The benchmark should not be used to evaluate agents in domains with different compliance requirements (e.g., medical advice, legal counsel) — the workflow_correctness rubrics are calibrated for B2B sales, not other professional domains.
